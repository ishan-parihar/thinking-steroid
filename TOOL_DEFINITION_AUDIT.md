# Tool Definition Audit: UX for AI Agents

**Date:** April 12, 2026
**Auditor:** Sisyphus (OhMyOpenCode)
**Subject:** 13 thinking modality tools — Do tool descriptions accurately signal behavior to consuming AI agents?

---

## Executive Summary

**Core Finding:** The tool definitions create a *perception mismatch* for consuming AI agents. The descriptions imply each tool produces a unique, specialized analytical output — but the reality is that all 13 tools are **thinking containers**: they provide structured epistemological frames for the agent's own reasoning, not data-fetching operations or externally-sourced analysis.

This is not inherently bad — structured thinking is valuable. But the current tool descriptions **over-promise uniqueness** and **under-communicate the container nature**, leading to:

1. **Tool proliferation anxiety** — agents believe they need to pick "the right tool" when in reality, most tools produce structurally similar outputs with different label permutations.
2. **Redundant invocation** — agents call multiple tools sequentially expecting novel information, when the marginal value of each additional tool diminishes rapidly.
3. **False confidence in outputs** — the structured format (tables, confidence scores, epistemic status) makes template-generated content appear authoritative.

**Severity:** Medium-High. This is a UX problem for the *consuming agent*, not a correctness problem. The tools work as designed — they just advertise themselves poorly.

---

## 1. The Fundamental Misalignment

### What the Agent Expects (from tool descriptions)

Each tool description reads like a specialized analytical service:

```
think_sequential: "Performs structured sequential thinking analysis... Each step includes a claim,
detailed reasoning, confidence level, underlying assumptions, a steel-manned counter-argument..."

think_causal: "Maps feedback loops (R/B), ranks intervention points 1-12, detects archetypes..."

think_shadow: "Multi-framework shadow analysis + shadow pattern detection + Jungian archetype matching..."
```

These descriptions read like: *"I will do specialized work for you."*

### What the Agent Actually Gets

Every tool runs the same underlying machinery:

1. **Template-based content generation** — pre-written prose blocks with variable substitution (`${pole}`, `${domain}`, `${problem}`)
2. **Content composer fallback** — attempts to extract domain-specific patterns from input text, falls back to templates if insufficient signal
3. **Hash-based determinism** — when templates need variation, uses string hashing to pick from pool (not true reasoning)
4. **Heuristic keyword matching** — Cynefin classification, bias detection, archetype matching all use keyword/regex pattern matching

**Every tool is a different lens on the same input.** No tool fetches external data, runs computation on real-world signals, or produces outputs that couldn't be generated by the agent's own reasoning — just structured differently.

## 2. Tool-by-Tool Definition Analysis

### 2.1 Over-Promising Tools (High Risk of Agent Confusion)

#### `think_shadow`
- **Description claims:** "Multi-framework shadow analysis + shadow pattern detection + Jungian archetype matching"
- **Reality:** Keyword-based template substitution. The 4 frameworks (Freudian, Jungian, Gestalt, Integral) each produce template paragraphs with domain words swapped in. Jungian archetype matching is string comparison against a fixed list.
- **Agent deception level:** HIGH — "multi-framework analysis" and "archetype matching" sound like sophisticated pattern recognition. They are string-matching against templates.
- **Recommendation:** Reframe as "Analyzes hidden assumptions and blind spots through 4 psychological frameworks. Output is a structured comparison — useful for surfacing what your reasoning may be missing."

#### `think_hierarchical`
- **Description claims:** "Developmental stage mapping across 8 stages x 4 quadrants + optional vision-logic substage analysis"
- **Reality:** Template-based stage descriptions with variable substitution. The "vision-logic substage analysis" is a lookup table with 4 entries.
- **Agent deception level:** HIGH — "developmental stage mapping" implies the tool can assess where a system sits on a developmental trajectory. It cannot — it generates stage descriptions and asks the user which fits.
- **Recommendation:** Reframe as "Provides a structured framework for assessing developmental stages. You supply the stage assessment; the tool generates stage-specific analysis across 4 quadrants."

#### `think_unity`
- **Description claims:** "6-subsystem multi-lens analysis with inter-system dialogue"
- **Reality:** The "inter-system dialogue" is pre-written exchange templates between subsystems (LoDS, LiDS, SoCS, DriS, QuaS, ShWS). The subsystems don't actually analyze — they produce templated output with domain words substituted.
- **Agent deception level:** HIGH — "inter-system dialogue" and "grand synthesis" suggest emergent insight from subsystem interaction. The dialogue is scripted.
- **Recommendation:** Reframe as "Generates a synthesis view across 6 analytical dimensions. Each dimension produces a perspective on your problem, and a structured dialogue format compares them."

#### `think_aqal_situational`
- **Description claims:** "4-quadrant analysis with strategies and 2nd-order effects + optional psychograph (7 lines of development)"
- **Reality:** Template paragraphs per quadrant. Strategies and 2nd-order effects are generic lists with domain words inserted. The psychograph is a fixed table structure.
- **Agent deception level:** MEDIUM-HIGH — "strategies" and "2nd-order effects" imply domain-specific recommendations. They are templated.
- **Recommendation:** Reframe as "Organizes analysis across 4 perspectives (individual/collective, interior/exterior). Provides a structured framework — you supply the domain-specific content."

### 2.2 Appropriately-Specified Tools (Low Risk)

#### `think_cynefin`
- **Description claims:** "Classifies situations using the Cynefin framework into Clear/Complicated/Complex/Chaotic/Disorder domains"
- **Reality:** This is honest — it does keyword-based classification with confidence scoring. The description accurately reflects what it does.
- **Agent deception level:** LOW — The tool says it classifies, and it classifies.
- **Recommendation:** Minor — add clarification that classification is keyword-based, not semantic.

#### `think_metacognitive`
- **Description claims:** "Ladder of Inference + 20-Bias Cognitive Audit"
- **Reality:** Actually implements regex-based bias detection and ladder-of-inference analysis with text extraction heuristics. More substantive than other tools.
- **Agent deception level:** LOW-MEDIUM — The bias detection is keyword-based but the ladder analysis actually parses the reasoning chain for structural markers.
- **Recommendation:** Clarify that bias detection uses pattern matching, not LLM-based analysis.

#### `think_sequential`
- **Description claims:** "Step-by-step reasoning with claims, confidence, assumptions, counter-arguments"
- **Reality:** Template-based but with mode-specific and sub-mode-specific differentiation. The 4 thinking modes (analytical/creative/critical/strategic) and 4 reasoning sub-modes (deductive/inductive/abductive/analogical) produce meaningfully different outputs.
- **Agent deception level:** LOW-MEDIUM — The templates are the most differentiated across modes of any tool.
- **Recommendation:** Minor — acknowledge that outputs are structured reasoning scaffolds, not empirical analysis.

### 2.3 Mixed Tools

#### `think_causal`
- **Description claims:** "Maps feedback loops, ranks intervention points, detects archetypes"
- **Reality:** Actually implements graph-theoretic cycle detection for feedback loops. Leverage points and archetype detection use keyword matching. The graph cycle detection is genuinely algorithmic.
- **Agent deception level:** MEDIUM — The feedback loop detection is real (DFS on adjacency graph). Archetype detection is keyword matching.
- **Recommendation:** Differentiate: "Algorithmically detects feedback loops via graph analysis. Archetype detection uses pattern matching against known structures."

#### `think_scenario`
- **Description claims:** "2x2 scenario matrix with archetypal narratives, pre-mortem, futures wheel"
- **Reality:** Scenario narratives and consequences are template pools selected by hash. Pre-mortem is a fixed list. Futures wheel is templated with Jaccard similarity deduplication.
- **Agent deception level:** MEDIUM-HIGH — "archetypal narratives" and "futures wheel" sound like generated content. They are template pools.
- **Recommendation:** Clarify that scenarios are generated from a curated template library, not free-form narrative generation.

#### `think_first_principles`
- **Description claims:** "Socratic decomposition + reconstruction from bedrock facts"
- **Reality:** Claim extraction is sentence splitting. Classification is keyword scoring. Socratic interrogation uses template pools selected by hash. The reconstruction blueprint is a fixed 8-step list.
- **Agent deception level:** MEDIUM — The claim classification has genuine heuristic logic. The Socratic interrogation is templated.
- **Recommendation:** Differentiate between the heuristic classification and the template-based interrogation.

#### `think_polarity`
- **Description claims:** "9-row polarity map + 16x4 integration spectrum + systems archetype detection"
- **Reality:** Template paragraphs with domain variable substitution. Archetype detection is keyword-based. The integration spectrum is a 16x4 grid of template cells.
- **Agent deception level:** MEDIUM — The polarity framework is a legitimate analytical structure. The content filling it is templated.
- **Recommendation:** Clarify that the tool provides a structured framework — the analytical value is in the structure, not the generated content.

#### `think_aqal_projection`
- **Description claims:** "Temporal projections across 4 quadrants at 3 time horizons"
- **Reality:** Template paragraphs per quadrant × time horizon.
- **Agent deception level:** MEDIUM — Same pattern as situational but with time dimension.
- **Recommendation:** Combine documentation with `think_aqal_situational` — they're the same engine with different parameters.

#### `think_navigator`
- **Description claims:** "DAG-based reasoning planning with dependency tracking and parallel execution"
- **Reality:** Actually builds dependency graphs and tracks state. The most "functional" tool — it orchestrates calls to other tools. But it doesn't execute them; it plans them.
- **Agent deception level:** LOW — The tool does what it says: builds a reasoning DAG. The agent must execute the nodes manually.
- **Recommendation:** Clarify that Navigator plans but does not execute — the agent must call each tool in the DAG.

## 3. Structural Problems Across All Tools

### 3.1 The Content Composer Illusion

Every tool attempts a "content composer" first, then falls back to templates:

```typescript
const composerAttempt = composeToolContent({...});
if (composerAttempt.length > 200) return composed;
return generateTemplateFallback(...);
```

**Problem:** The agent has no visibility into which path was taken. A 200-character threshold means that marginal input produces template output indistinguishable from composed output in format. The agent receives structurally identical markdown regardless of whether the content was domain-specific or templated.

**Impact:** The agent cannot know if its output is rich or thin. This is the core UX problem — the tool's output format masks content quality.

### 3.2 Output Format Uniformity

Every tool produces the same output envelope:
- Markdown headers
- Bullet-point lists
- Confidence/epistemic status section
- Suggested follow-ups

This uniformity makes all tools *feel* equivalent to the agent, even when some (like `think_causal`'s graph cycle detection) are algorithmically richer than others (like `think_shadow`'s template substitution).

### 3.3 Epistemic Status Inflation

Every tool calculates an "epistemic status" (well-supported/tentative/speculative) based on trivial metrics:
- Input length
- Depth parameter
- Whether optional params were provided

This creates a false sense of epistemic rigor. A 500-character input to `think_first_principles` gets "tentative" status not because the analysis is tentative, but because the input was short.

### 3.4 Suggested Follow-Ups Are Circular

Every tool suggests running other tools in this same server. This creates an infinite loop potential:
```
think_sequential → suggests think_aqal_situational
think_aqal_situational → suggests think_shadow
think_shadow → suggests think_unity
think_unity → suggests think_sequential
```

The agent may burn tokens cycling through tools without gaining new information.

## 4. Recommendations

### 4.1 Critical: Rewrite Tool Descriptions for Honesty

**Current pattern:** "Performs X analysis using Y framework. Best for Z."
**Recommended pattern:** "Structures your reasoning around X framework. Does NOT fetch external data or perform empirical analysis — it organizes YOUR input through a structured lens. Best for Z."

Example rewrite for `think_shadow`:
```
Analyzes hidden assumptions and blind spots through 4 psychological frameworks (Freudian, Jungian, Gestalt, Integral).

What this tool DOES: Organizes your input through structured frameworks that surface what your reasoning may be missing. Each framework highlights different types of blind spots.

What this tool does NOT: Access external data, read minds, or detect actual psychological patterns. All analysis is derived from the text you provide.

Best for: Surfacing assumptions your reasoning takes for granted, identifying what perspectives you haven't considered, and creating structured blind-spot inventories.
```

### 4.2 Add Output Quality Indicators

Every tool response should include:
```
## Output Quality Note
- Content source: [Template-based | Composer-enhanced | Hybrid]
- Domain specificity: [Low | Medium | High] — based on input length and keyword density
- Recommended: [Provide more context for better analysis | Analysis is well-grounded in provided input]
```

### 4.3 Consolidate Redundant Tools

| Current | Relationship | Recommendation |
|---------|-------------|----------------|
| `think_aqal_situational` + `think_aqal_projection` | Same engine, different params | Merge into single `think_aqal` with `mode: situational | projection` |
| `think_polarity` + `think_causal` (archetype detection) | Both detect Senge archetypes | Share archetype detection code; clarify in descriptions |
| `think_sequential` + `think_navigator` | Navigator calls sequential | Clarify that Navigator is a planner, sequential is an executor |

### 4.4 Reduce Suggested Follow-Ups

Instead of suggesting 3-4 tools, suggest **one** tool with a reason:
```
## Suggested Next Step
Run `think_metacognitive` to audit the reasoning quality of this analysis before acting on its conclusions.
```

### 4.5 Add a "What Is This?" Section to Server Instructions

The `INSTRUCTIONS` string in `index.ts` should include:
```
IMPORTANT: This server provides STRUCTURED THINKING TOOLS — not data retrieval or empirical analysis tools. 
Each tool reorganizes the information YOU provide through a different analytical lens. 
No tool fetches external data, performs computation on real-world signals, or produces 
outputs that contain information beyond what you provide as input.

Use these tools when you want to:
- Structure your reasoning in a specific way
- Surface assumptions or blind spots in your thinking
- View a problem from multiple perspectives
- Plan a multi-step analytical workflow

Do NOT use these tools when you want to:
- Fetch data from external sources
- Perform statistical analysis on datasets
- Get empirically-validated answers
```

### 4.6 Implement Output Truncation Awareness

Several tools produce outputs that exceed practical consumption limits (47KB+ for polarity, 66KB+ for projection). The `enforceLimit` function truncates at `CHARACTER_LIMIT` but the agent doesn't know it was truncated. Add explicit truncation markers:

```
⚠️ Output truncated at {CHARACTER_LIMIT} characters. 
For full output, use output_depth="essential" for summary or provide more focused input.
```

## 6. Conclusion

The thinking-steroid tools provide a legitimate service: **forcing different reasoning topologies on AI agent thinking**. The problem is not the tools themselves — it's that their descriptions promise specialized analysis when they deliver structured reframing.

The consuming agent would benefit enormously from:
1. **Honest descriptions** that clarify these are thinking containers, not data sources
2. **Quality indicators** in output that reveal whether content was composed or templated
3. **Consolidation** of tools with overlapping functionality
4. **Guard rails** on suggested follow-ups to prevent infinite analysis loops

The most critical single change: **rewrite every tool description to lead with what the tool DOESN'T do** (fetch data, perform empirical analysis, access external information) before describing what it DOES (structure reasoning through a specific framework).
